# How to use this repository for development

## Setup

**Python 3.10–3.13 supported** (3.11 recommended for development).
Audio packages (`soundfile`, `yakyak`) have wheels for all supported Python
versions on macOS ARM — there is no 3.12+ ceiling. Use `python3.11` in the
venv example below, but any 3.10–3.13 works equally well.

```bash
# 1. Install Python 3.11 if not already present
brew install python@3.11

# 2. Clone both repos independently
git clone https://github.com/b202i/cross-st.git
git clone https://github.com/b202i/cross-story.git
ln -s ~/github/cross-story ~/github/cross/story
cd cross

# 3. Create the venv with Python 3.11
python3.11 -m venv .venv
source .venv/bin/activate

# 4. Install dependencies
pip install -r requirements.txt

# 5. Symlink st-* commands into the venv (run AFTER pip install)
bash script/symbolic_links.bash
```

The symlinks land in `.venv/bin/` and are available on `PATH` whenever the venv
is active. The script is safe to re-run (e.g. after adding a new `st-*.py` file).

For full install instructions see [README_install.md](README_install.md).

The fact-check score represents the average evaluation of all statements in a report. Each statement is scored as follows: 2 points for true, 1 point for mostly true, 0 points for opinion (excluded from the average), -1 point for mostly false, and -2 points for false. For every statement in the document, the AI will restate it and provide a detailed explanation justifying the assigned score.

3. Present the result of a cross product of AI on AI
results:

ca, cb, cc

### Data storage
Grok and OpenAI have different response formats.
For story data, the app will maintain an array of story data for each AI.
The app also maintains an array of processed story products.

```code
    data = {
        "make": args.ai,
        "model": ai_model,
        "gen_payload": gen_payload,
        "gen_response": gen_response,
    }
    story = {
        "make": select_make,
        "model": select_model,
        "title": file_title_content,
        "markdown": file_md_content,
        "text": file_txt_content,
        "spoken": as_spoken,
        "hashtags": select_hashtags,
    }
    container = {
        "story": story,
        "data": data,
    }
```
With this approach you can have as many AI entries as you like, with the same make, or all from different makes, any combination. You could even have the exact same make and model, but with different parameters.

You can also have as many processed stories as you like.

+ + and AI fact checking in the domain of XYZ. The goal is to determine AI effectiveness 
+ + in report generation and fact-checking while exploring bias issues.
+ The preamble is injected into the AI prompt for added context, and is
+ used in the introduction of the report for human context.

  + st-plot create ./tmp if not exists

    + plots[plot-type-data] = plot-data 
    + plots[plot-type-path] = plot-path
    + upload the plot to the target discourse server
    + plots[plot-type-url] = plot-url
2. st-analysis builds an analysis prompt/w plot data
  + preamble instructions for the analysis
  + for each plot-type, create a sub-prompt
    + Produce a caption for this {plot-type} data
    + The user will be shown this data in the form of a plot
    + What does the data reveal? What should a reader take away from this data.
    + Use a full sentence to a full paragraph for a caption for the data.
    + place the caption directly after this url {plot-type-url}
2. st-analysis generates report
  + "data" is updated similar to any other ai generated story
  + "data" is updated with plot info: in-memory-json
  + "story" is added

# Bang story generation
This is documentation for a speedup for the ST AI command line system.
Main idea, fire off AI story generation in parallel, merge results.
## Current serial operation
This example is with 4 AI, the system can use a single AI or even 10 different AI.
Currently the ST system works in a serial fashion.
1. Instruct a specific AI to produce a report from a prompt.
2., 3., 4. Repeat this for each AI.
Results are saved in a json array, [0:3].
5. The 5th AI call merges the previous 4 reports creating a 5th report, [4].
## Bang Concept
1. Fire off the initial 4 AI calls in parallel, each with a separate AI
2. Merge the results from 1., fire off AI call to merge the 4 reports.
## Performance increase expected
The current serial operation requires the time of 5 AI calls.
The Bang concept takes the time of 2 AI calls.
## Data storage and merging
To keep things tidy, the ST system uses 2 files, a .json JSON file,
and a text file for the .prompt text file. Example: subject.json, subject.prompt.
At the top of the JSON file, there is an array of objects under "data", 
and another array of objects under "story".
The first data object is ["data"][0], and the second is ["data"][1], etc.
and the first story object is ["story"][0], and the second is ["story"][1], etc.
## Bang story generation
Generate a story from each AI using the same prompt, into separate .json files.
st-bang starts the story generation for each AI, while monitoring results:
```bash
st-bang --ai all subject.prompt
```
Starts processes:
    st-gen --ai xai subject.prompt --bang 0 (creates subject_0.json)
    st-gen --ai openai subject.prompt --bang 1
    st-gen --ai anthropic subject.prompt --bang 2
    st-gen --ai perplexity subject.prompt --bang 3
A blocking file is created for each st-gen process:
    subject_0.json.block
    subject_1.json.block
    subject_2.json.block
    subject_3.json.block
When each st-gen processes completes, the .block file is removed
st-bang polls every few seconds
When all .block files are removed, st-bang integrates the subject_n.json files
into a single subject.json, using the ["data"] and ["story"] arrays
## Create the final merged story
After all of the .block files are removed and the data and story are intetegrated
back into a single .json file, st-merge is called to make the final story
> st-merge --stories 1 2 3 4 -- subject.json
The bang process is complete
